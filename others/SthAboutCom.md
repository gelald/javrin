# 老板篇

## 对内

1. 项目的立项，应该有一定的资金与有一定技术能力的伙伴。首先资金这一方面我没有过多地去询问，我说说技术伙伴这一层面。自己一方面又想把握项目核心，一方面又想省钱，怕招进来的人不好用，所以他最开始找的人，他是南网旗下一家外包公司的小领导（月薪也有好几万），同时又是他老婆认识很久的同学，合作的方式：口头协议，每月给几千块钱工资，这个项目让他来做。但是他负责项目的三四年来一直没有想过要把他挖过来，确立为公司的核心技术人员，却不断地从学校里面找学生来接触项目，想着能够从中找到能力好的并且最终确立为核心人员，一个月补贴一千块钱，加之创业公司吸引力也小，最终许多学生在这个项目中进进出出，离开了的然后就说人家没有沉下心来学习技术。
2. UI姐姐。一开始是有一个很好的苗子，设计能力非常好，为了一个配色，专门去问了很多人的对某几个颜色的接受能力，算是做了小调研，现在我们几个客户端的UI都是她设计的，但最终也还没有留下来。后续多次提及需要UI，但他一直说没能找到和之前那个小姐姐那样有能力的，一直抱着一个想法：又要招到后马上能很好的完成工作，又不愿意花钱，导致后面有些地方的UI着实有点难接受。
3. 接项目做。本来开发人员就三个人，还接外面的项目，三句不离两句就说能不能快速地把他完成，把接来的项目描述得多简单，随便弄弄有个样子就好。我认为尽管项目有多简单，要求有多捡漏，我们的重心都应该放在自己的项目上面，因为我们现在开发力量捉襟见肘，不应该再分心去弄其他无关紧要的项目，除非我们的工作团队壮大了，有余力了，那么可以去接这些项目来赚赚快钱。
4. 拖欠工资
5. 一开始不去找一个相对靠谱的技术领导人(cto)，宁肯把产品核心技术、架构外包给别人(兼职形式，没有合同)。虽然说自己也会去招人，但是招人的套路却是：因为我们团队其中一个合伙人是我大学的班主任，他就到处找同样是老师的关系好的朋友发一些招人信息。招聘的要求也是兴趣为导向，只要你有兴趣，就可以参与，比很多学校工作室的招人门槛还低，并且招进来的学生都能快速上手，导致人员进进出出，核心团队一直没有完成扩展升级，还老抱怨这些学生心气太高傲，没办法沉下心来学习。每周的会议是非常重要的，既是对本周工作的总结，又是对下周工作的安排，每次会议我都有做记录，但是同一个功能这周说要做，下周又改说法说不用做，再下一周又说要做，反问他，他就马上否认。技术领导每次过来开会，绝大部分时间在玩手机，我们项目需要用到他们公司开发的一些基础服务，其中确实论证了在本系统中是有严重缺陷的，也不修改，所以很多东西只能不断迁就他们的服务来做，导致系统逐渐畸形，维护难度变大。

## 会议

1. 需求想当然，开会的时候想法天马行空，并且还想着任何需求都能能做做一版出来。一个例子：前提条件是开发人员3个，服务器资源本身就非常吃紧，带宽是最低配置，疫情期间，本想着按计划完成我们之前想好的业务继续往下做，突然会议里面说想做个直播，让我们去网上找人家做好的工具拿过来用，快速出一版。又比如：我们做的一个图文编辑工具，做好了一个需求（定义几个人物，定义他们说的话或者音频，到客户端的时候做一个自动说话的功能），突然说这个不太好用，想要做一个像qq秀一样的东西，不断地给一个人物拼接上衣服裤子各种装饰。其实我觉得这样有点舍本逐末了，并且还要求做个简单的3D的人物快速出一版，着实佩服。
2. 颠覆需求。每次按照要求做好之后，总想着去修改，修改其实没关系，重要的是大修改，并且还说一开始说的就是这样的，我们都有做会议记录的，总是回应他说一开始说的不是这样子的，他也完全否认自己说过的内容，导致一次工作花费五六次时间。
3. 那一次要开始研发数学内容，说好调研，那个月每次开会都有问他调研的进度怎么样，每次都提出一个确切的日期，但是最终都无法落实到位。其实没做完可以直接说在做的，不需要每次都说下周能搞定。而且调研的结果并非是一份文档，仅仅是一个视频。这个视频还只是他认识的一个学校里面有个年级用的一套远大教育的数学内容，和我们一开始设计的图文编辑工具能做出的内容完全不是一个类型，非要生搬硬套，最终还是确立这就是以后要做的数学内容，说好调研，没想到是抄袭，而且是不考虑已有系统的抄袭。

# 技术领导篇

## 工作

1. 我们系统的架构是微服务架构，技术选型完全按照目前南网项目的技术架构，问他选择这个组件的原因，就说我们南网用的是这一套。而且项目的搭建完全从以前做过的项目copy过来，配置文件、通用类中充斥着大量的旧项目的注释，甚至还放了不少无关的配置文件，令新进来的小伙伴包括我，无法全面地理解项目。
2. 他一周能来一次，就是开会。现在开始就把工作都交给我了，自己公司工作开始变多，变忙碌，工作时间又缩减，加之老板每个月都拖欠工资，所以更不想做这边的工作，老板又觉得他不干活，更不想发，形成死循环了。现在他过来开会都只是吹吹牛逼，剩下难的工作都交给我来做，问他他也时常不回复，有比较多的技术问题比较难突破，他都不怎么理我，既然是我的技术领导，应该需要为我提供一定的帮助。

# 收获篇

## 领导能力

1. 把握进度能力，当有机会带领新人时，不一定需要你的知识面能够完全包围他的知识面，当然能做到这一点说明你知识掌握度非常优秀。当分配到工作给下属的时候，你需要结合功能复杂度和个人能力把控到这个功能的完成时间，这样项目的进度才能正常有序地向前进行，否则，后续的工作就无法继续安排了，只能被迫进入「阻塞」状态。
2. 文档能力，安排工作的时候，不是简单的三言两语就可以完成的，而且也不规范，有能力的话还是拟出完整文档，必要时带上原型图、UML图、流程图等等，这样也方便下属，避免疏漏导致重复问同样问题的情况
3. 揽责能力（背锅能力），正式生产环境中出现的问题，尽管是下属一手造成的，也不能置之度外，甚至退一步来讲，没有把关好提交的程序，也是我的责任。应该做一份好的夹心，面对上级领导需要合适地揽责，面对下属需要有针对性的教导，比如程序测试时要求他们出一份测试报告等等。
4. 请客这一方面的话。在正式入职之前没有一个月工资收入超过3500，但是我在正式入职前的工作时间是23个月，买件衣服，算上吃饭交通，一个月基本上能存个几百到一千块钱，当时是想存存钱换电脑的，请客这方面，我算是铁公鸡了，因为实在是拮据。
5. 其他，项目规范能力，仔细研读阿里巴巴开发规范，命名的规范需要铭记于心，而且英语单词要使用恰当，记得那一次我让其中一个下属完成一个关于题目的解析项设置，半夜恰好有空看了看他的commit，题目的解析应该使用「resolution」而他使用了解析数据用的「parsing」，连夜连续他修改代码并提交。
