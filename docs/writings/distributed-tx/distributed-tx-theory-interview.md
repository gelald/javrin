## 高频面试题 Top 10

### Q1：什么是 CAP 定理？为什么分布式系统不能三者兼顾？

**标准回答**：

CAP 定理指出分布式系统有三个核心特性：一致性（C）、可用性（A）、分区容忍性（P）。由于网络分区是不可避免的客观现实（网络抖动、光缆中断），P 是必须选择的。所以在发生网络分区时，只能在 C 和 A 之间二选一。

- **CP 系统**：保证一致性，牺牲可用性。如 ZooKeeper 在 Leader 选举期间不可用。
- **AP 系统**：保证可用性，牺牲一致性。如 Eureka 节点间异步复制，始终可响应。

> **加分点**：实际系统中 CAP 不是非此即彼的，同一个系统在不同模块可以有不同的选择。比如核心交易用 CP，商品展示用 AP。

---

### Q2：什么是 BASE 理论？和 ACID 有什么关系？

**标准回答**：

BASE 是对 CAP 中 AP 方向的工程实践总结：
- **BA（Basically Available）**：基本可用，允许响应时间变长或功能降级
- **S（Soft State）**：软状态，允许中间状态（如"处理中"）
- **E（Eventually Consistent）**：最终一致性，一段时间后数据最终一致

BASE 是对 ACID 强一致性的放松。ACID 适合金融转账等核心场景，BASE 适合互联网高并发场景。实际项目中通常是混合策略——核心链路用 ACID，非核心链路用 BASE。

---

### Q3：2PC 的原理是什么？有什么问题？

**标准回答**：

2PC（两阶段提交）分两个阶段：
1. **Prepare 阶段**：协调者向所有参与者发送 prepare 请求，参与者执行事务但不提交，写入 redo/undo log，返回 Yes/No
2. **Commit 阶段**：所有参与者都 Yes 则 commit，任一 No 则 rollback

**三大问题**：
1. **同步阻塞**：Prepare 后参与者一直持有锁直到 Commit，高并发下性能极差
2. **单点故障**：协调者宕机，所有参与者永远阻塞
3. **数据不一致**：阶段二只有部分参与者收到 commit

---

### Q4：XA 协议了解吗？为什么实际项目中很少用？

**标准回答**：

XA 是 X/Open 组织定义的分布式事务标准，本质是 2PC 的规范实现。涉及三个角色：
- **AP**（应用程序）、**TM**（事务管理器/协调者）、**RM**（资源管理器/参与者）

Java 中通过 JTA 接口实现，常用 Atomikos 作为 TM。

**很少用的原因**：
1. **性能差**：持有锁时间长，不适合高并发
2. **不适合微服务**：跨服务走 RPC，XA 只能协调同一进程内的多数据源
3. **运维复杂**：TM 单点问题需要额外解决
4. **不是所有中间件都支持**：比如 Redis、MongoDB 早期版本不支持 XA

---

### Q5：3PC 了解吗？和 2PC 的区别？

**标准回答**：

3PC 把 2PC 的 Prepare 阶段拆成两步：CanCommit（询问）和 PreCommit（预提交），并引入超时机制。

**改进点**：参与者超时后可以自动提交，减少了阻塞。

**新问题**：超时自动提交可能与协调者的回滚意图冲突，导致数据不一致。

**结论**：3PC 工程上几乎没有使用，更多是理论意义。实际项目中要么用 2PC/XA（同进程多数据源），要么直接用最终一致性方案（跨服务场景）。

---

### Q6：CAP 在实际系统中是怎么取舍的？举例说明。

**标准回答**：

| 场景 | 选择 | 原因 |
| --- | --- | --- |
| 金融转账 | CP | 数据不能错，宁可不可用 |
| 商品展示 | AP | 大促时宁可展示旧数据也不能白屏 |
| 服务注册发现 | AP（Eureka） | 注册中心不可用比数据不一致影响更大 |
| 分布式锁 | CP（ZooKeeper） | 锁数据不一致会导致并发问题 |

> **加分点**：大部分互联网系统在大部分时间网络是正常的（没有分区），C 和 A 可以同时满足。CAP 的取舍主要体现在**网络分区发生时**的应急策略。

---

### Q7：什么是分布式事务？你们项目是怎么解决的？

**标准回答**：

分布式事务是指跨多个服务/数据库的事务操作，需要保证这些操作要么全部成功，要么全部回滚。

主流方案有四种：
1. **TCC**：Try-Confirm-Cancel，适合资金类强一致场景
2. **Saga**：正向操作 + 补偿操作，适合长流程业务编排
3. **本地消息表**：实现简单，适合低并发场景
4. **RocketMQ 事务消息**：高性能，适合异步解耦场景

> **面试策略**：这个问题面试官期待你结合项目经验回答。可以说"我们用的是 RocketMQ 事务消息 + 本地消息表兜底"，然后展开讲。

---

### Q8：强一致性和最终一致性有什么区别？什么时候用哪个？

**标准回答**：

- **强一致性**：写入完成后，任何后续读取都能看到最新值。实现成本高，性能差。
- **最终一致性**：写入后暂时允许读到旧值，但最终（毫秒到秒级）所有副本会一致。

**选型原则**：
- 涉及**资金**的操作（余额变更、扣税）→ 强一致
- 涉及**状态同步**的操作（订单状态通知、积分变更）→ 最终一致
- 涉及**数据分析**的操作（报表统计、日志汇总）→ 最终一致

---

### Q9：为什么说 2PC 不适合微服务架构？

**标准回答**：

1. **跨服务调用走 RPC**：2PC/XA 只能协调同一进程内的多数据源，无法跨服务协调
2. **同步阻塞与微服务理念冲突**：微服务强调高可用、异步解耦，2PC 的同步阻塞背道而驰
3. **服务自治被破坏**：2PC 要求所有参与者听从协调者指令，微服务的独立部署和演进受到限制
4. **故障放大**：一个服务的数据库问题可能阻塞整条调用链

微服务架构中更推荐 TCC、Saga、消息驱动等最终一致性方案。

---

### Q10：如果面试官问"你了解哪些分布式事务方案"，你怎么组织回答？

**推荐回答框架**：

> 分布式事务方案我了解四种主流方案：
>
> **第一类是强一致性方案**，即 2PC/XA。通过协调者统一管理参与者的提交和回滚，但同步阻塞问题严重，实际项目中很少使用。
>
> **第二类是 TCC**，把业务拆成 Try/Confirm/Cancel 三步，适合资金类对一致性要求高的场景，但业务侵入性强，需要为每个操作写三个方法。
>
> **第三类是 Saga**，把长事务拆成多个本地事务，每个本地事务配一个补偿操作，适合业务流程长的场景。
>
> **第四类是基于消息的最终一致性**，包括本地消息表和 RocketMQ 事务消息。性能最好，实现相对简单，适合大多数互联网场景。
>
> 我们项目中用的是 RocketMQ 事务消息方案，核心链路用事务消息保证最终一致，同时有定时任务兜底补偿。

## 易错点总结

| 易错点 | 正确理解 |
| --- | --- |
| "CAP 是三选二" | P 必选，实际是 C 和 A 二选一 |
| "BASE 就是不要一致性" | BASE 是"最终"一致，不是"不要"一致 |
| "2PC 可以解决所有分布式事务问题" | 2PC 只适合同进程多数据源，不适合跨服务 |
| "3PC 比 2PC 好" | 3PC 缓解了阻塞但引入了新的不一致问题，几乎没有实际使用 |
| "XA 就是分布式事务的最佳实践" | XA 性能差、不适合微服务，实际很少用 |
| "CAP 要么 CP 要么 AP" | 系统正常运行时 C 和 A 可以同时满足，CAP 取舍只在网络分区时 |
