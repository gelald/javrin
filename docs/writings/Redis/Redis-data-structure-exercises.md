## 一、选择题（每题 5 分，共 25 分）

### 题目 1
**关于 SDS（Simple Dynamic String）与 C 字符串的区别，以下说法错误的是？**

A. SDS 获取字符串长度的时间复杂度为 O(1)，C 字符串为 O(n)
B. SDS 的空间预分配策略可以减少内存重分配次数
C. SDS 通过 `\0` 判断字符串结束，因此不是二进制安全的
D. SDS 在进行追加操作时，会先检查空间是否足够，不够则自动扩容

<details>
<summary>答案与解析</summary>

**答案：C**

SDS 是二进制安全的。C 字符串通过 `\0` 判断字符串结束，因此不能存储包含 `\0` 的二进制数据。而 SDS 使用 `len` 字段记录字符串长度，不依赖 `\0` 来判断结束，因此可以存储任意二进制数据。SDS 也会在末尾保留 `\0`，目的是为了兼容部分 C 字符串函数。

**面试话术**：SDS 相比 C 字符串有三大优势——O(1) 获取长度（`len` 字段）、杜绝缓冲区溢出（API 先检查再操作）、二进制安全（用 `len` 而非 `\0` 判断结束）。

</details>

### 题目 2
**关于 Redis 字典（Dict）的渐进式 rehash，以下说法正确的是？**

A. rehash 期间所有操作都只在 ht[1] 上执行
B. rehash 是一次性将所有键值对从 ht[0] 迁移到 ht[1]
C. rehash 期间，查找操作会先查 ht[0]，再查 ht[1]
D. 负载因子小于 0.1 时触发扩容 rehash

<details>
<summary>答案与解析</summary>

**答案：C**

渐进式 rehash 的核心规则：
- **查找（CRUD 的读）**：先查 ht[0]，找不到再查 ht[1]
- **新增（C）**：只写入 ht[1]，保证 ht[0] 的键只会减少不会增加
- **修改（U）/ 删除（D）**：如果在 ht[0] 中找到，迁移到 ht[1] 后再操作

选项 A 错误，rehash 期间 CRUD 操作涉及两个哈希表；选项 B 错误，渐进式 rehash 就是分多次、逐步完成的，避免一次性大量操作阻塞服务；选项 D 错误，负载因子 >= 1 且没有执行 `BGSAVE`/`BGREWRITEAOF` 时扩容，负载因子 < 0.1 时收缩。

**面试话术**：渐进式 rehash 将一次性的大量迁移分散到每次 CRUD 操作中，每次迁移 ht[0] 的一个索引位置上的所有键值对，避免阻塞 Redis 主线程。

</details>

### 题目 3
**Redis 为什么使用跳表而不是红黑树来实现有序集合（ZSET）？**

A. 跳表的查找时间复杂度优于红黑树
B. 跳表实现更简单，且范围查询更高效
C. 跳表的内存占用一定比红黑树少
D. 红黑树不支持范围查询

<details>
<summary>答案与解析</summary>

**答案：B**

跳表与红黑树的查找时间复杂度都是 O(log n)，选项 A 错误。跳表的每个节点有多个前向指针，内存占用通常略高于红黑树，选项 C 错误。红黑树也可以范围查询，只是实现更复杂，选项 D 错误。

Redis 作者 antirez 的原话大意：跳表与平衡树性能相当，但实现更简单；而且跳表的范围查询更自然——找到起点后沿底层链表遍历即可，不需要像平衡树那样进行中序遍历。

**面试话术**：跳表 O(log n) 查找 + O(n) 范围扫描，实现简单（比红黑树好维护），范围查询只需沿底层链表顺序遍历，天然适合 ZRANGEBYSCORE 这类操作。

</details>

### 题目 4
**关于整数集合（IntSet）的编码升级，以下说法正确的是？**

A. 当向 int16 类型的 IntSet 中插入一个 int32 值时，只会升级该元素所在的槽位
B. IntSet 支持编码降级，当删除大值后可以回退到较小的编码
C. IntSet 升级时会将所有现有元素都转换为新的编码类型
D. IntSet 最多只能存储 int32 类型的整数

<details>
<summary>答案与解析</summary>

**答案：C**

IntSet 编码升级的核心过程：当新插入的整数类型大于当前编码时，会先将整个数组的编码升级（int16 → int32 → int64），所有现有元素都转换为新的编码类型，然后再插入新元素。

选项 A 错误，升级是整个数组级别的操作；选项 B 错误，IntSet 不支持降级，这是面试高频考点；选项 D 错误，IntSet 支持 int16、int32、int64 三种编码。

**面试话术**：IntSet 的编码升级是"一升全升"，且不支持降级。设计上保证元素有序、编码统一，牺牲了灵活性换取内存紧凑。

</details>

### 题目 5
**关于 String 类型的编码转换，以下说法正确的是？**

A. 一个 embstr 编码的字符串对象，执行 APPEND 命令后仍然是 embstr 编码
B. 当字符串值长度超过 44 字节时，编码会从 embstr 转换为 raw
C. 当字符串值可以用 long 类型表示的整数时，编码为 embstr
D. int 编码的字符串对象在执行 APPEND 追加非数字字符后，编码保持不变

<details>
<summary>答案与解析</summary>

**答案：B**

String 类型三种编码的转换规则：
- **int**：值为整数且可以用 long 表示 → 直接存放在 ptr 指针位置（不用额外分配内存）
- **embstr**：值长度 <= 44 字节 → redisObject 和 SDS 一次内存分配，紧凑存储
- **raw**：值长度 > 44 字节 → redisObject 和 SDS 分两次分配

选项 A 错误，embstr 是只读的，任何修改操作（如 APPEND）都会将其转换为 raw 编码；选项 C 错误，整数值的编码是 int 而非 embstr；选项 D 错误，int 编码追加非数字字符后会转换为 raw 编码。

**面试话术**：44 字节是临界点（Redis 3.2+），embstr 只读且紧凑（一次分配），raw 可修改但两次分配。面试常问"embstr 为什么只读"——因为 redisObject 和 SDS 连续存储，修改后可能需要 realloc，破坏了紧凑性。

</details>

## 二、填空题（每空 2 分，共 20 分）

### 题目 6
SDS 的空间预分配策略：当 SDS 修改后长度小于 1MB 时，会分配与 `len` 属性__________的未使用空间；当修改后长度大于等于 1MB 时，会分配__________的未使用空间。

<details>
<summary>答案与解析</summary>

**答案**：相等、1MB

SDS 空间预分配规则：
- `len < 1MB`：`free = len`（额外分配和已使用长度相等的未使用空间）
- `len >= 1MB`：`free = 1MB`（额外分配固定 1MB 的未使用空间）

这样设计的目的是在连续增长字符串时减少内存重分配次数，兼顾空间与时间效率。

</details>

### 题目 7
Redis 字典使用__________算法解决哈希冲突，当负载因子达到__________时（且没有执行 BGSAVE/BGREWRITEAOF），触发扩容 rehash。

<details>
<summary>答案与解析</summary>

**答案**：链地址法（或拉链法）、1

Redis 字典采用链地址法解决哈希冲突，每个桶是一个链表。触发扩容的条件：
- 负载因子 >= 1 且没有在执行 `BGSAVE` 或 `BGREWRITEAOF`
- 负载因子 >= 5（强制扩容，无论是否在执行持久化）

触发收缩的条件：负载因子 < 0.1

之所以在 BGSAVE/BGREWRITEAOF 时不扩容，是因为子进程使用写时复制（Copy-On-Write），此时扩容会创建大量新页面，浪费内存。

</details>

### 题目 8
跳表每个节点的层高是随机生成的，层数范围为__________层到__________层。层数越高的节点越__________（填"多"或"少"）。

<details>
<summary>答案与解析</summary>

**答案**：1、32、少

跳表层高生成规则：每次创建节点时，以 1/4 的概率决定是否继续增加层数，最大 32 层。层数越高的节点概率越低，呈指数衰减，类似于理想情况下的平衡树结构。

查找复杂度 O(log n) 的本质：高层节点作为"快速通道"跳过大量中间节点，类似于二分查找的思路。

</details>

### 题目 9
压缩列表（ListPack）中每个 entry 记录的是__________的长度（backlen），而旧版 ZipList 每个 entry 记录的是__________的长度（prevlen）。这一改变从根本上解决了__________问题。

<details>
<summary>答案与解析</summary>

**答案**：自身、前一节点、连锁更新

ListPack 的 entry 结构为 encoding + data + backlen，其中 backlen 记录的是自身长度。而 ZipList 的 prevlen 记录前一节点长度，变长编码（< 254 用 1 字节，>= 254 用 5 字节）可能导致连锁更新。ListPack 记录自身长度后，修改某个 entry 不会影响其他 entry 的存储格式。

</details>

### 题目 10
redisObject 的五个字段分别是：type（__________）、encoding（__________）、ptr（__________）、lru（__________）、refcount（__________）。

<details>
<summary>答案与解析</summary>

**答案**：
- type：数据类型（string / list / hash / set / zset）
- encoding：编码方式（底层数据结构的实现方式）
- ptr：指向底层数据结构的指针
- lru：记录最后一次被访问的时间（用于内存淘汰策略）
- refcount：引用计数（用于内存回收和对象共享）

面试常考：通过 `OBJECT ENCODING key` 可以查看编码方式，通过 `TYPE key` 查看数据类型。一个 type 可以对应多种 encoding，这是 Redis 灵活节省内存的关键设计。

</details>

## 三、简答题（每题 10 分，共 30 分）

### 题目 11
**请解释 Hash 类型从 listpack 编码转换为 hashtable 编码的触发条件是什么？为什么 Redis 要设计两种编码方式？**

<details>
<summary>答案与解析</summary>

**转换条件**（满足任一即转换）：

1. 哈希对象中所有键值对的键和值的字符串长度都 <= 64 字节，且键值对数量 <= 512 个 → 使用 listpack
2. 任意键或值的字符串长度 > 64 字节，或键值对数量 > 512 个 → 转换为 hashtable

这两个阈值可以通过 `hash-max-listpack-value`（默认 64）和 `hash-max-listpack-entries`（默认 512）配置。

**设计两种编码的原因**：

- **listpack**：内存紧凑、连续存储、缓存友好。适合元素少且短的场景（如用户配置信息），节省大量内存开销。缺点是修改操作可能涉及内存重分配。
- **hashtable**：O(1) 时间复杂度的增删改查，适合元素多的场景。缺点是哈希表本身有指针开销和装载因子带来的额外空间浪费。

这是 Redis "用 CPU 时间换内存空间"设计哲学的体现。

</details>

### 题目 12
**请简述 QuickList 的设计思路。为什么它是 listpack 和 linkedlist 的折中方案？**

<details>
<summary>答案与解析</summary>

**QuickList 的设计思路**：

QuickList 是一个由 listpack 节点组成的双向链表。每个链表节点内部是一个 listpack，存储一批连续的数据元素。

```
+----------+     +----------+     +----------+
| listpack | <-> | listpack | <-> | listpack |
| (一批元素) |     | (一批元素) |     | (一批元素) |
+----------+     +----------+     +----------+
```

**为什么是折中方案**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 纯 linkedlist | 操作灵活，O(1) 插入删除 | 每个节点前后指针占 16 字节，内存碎片多 |
| 纯 listpack | 内存紧凑连续 | 元素过多时 realloc 开销大 |
| QuickList | 兼顾内存紧凑和操作效率 | 实现复杂度增加 |

QuickList 通过控制每个 listpack 节点的大小（`list-max-listpack-size` 配置），在内存和性能之间取得平衡。还可以通过 `list-compress-depth` 对中间节点进行 LZF 压缩，进一步节省内存。

</details>

### 题目 13
**请列举五种基本数据类型（String、Hash、Set、List、Zset）各自的编码转换条件，并说明 Redis 为什么要进行编码转换。**

<details>
<summary>答案与解析</summary>

**五种类型的编码转换条件**：

| 类型 | 小数据编码 | 大数据编码 | 转换临界条件 |
|------|-----------|-----------|-------------|
| String | int / embstr | raw | 整数 → int；<= 44 字节 → embstr；> 44 字节 → raw；embstr 修改后 → raw |
| Hash | listpack | hashtable | 元素数 > 512 或字段/值长度 > 64 字节 |
| Set | intset / listpack | hashtable | intset: 不全为整数 或 元素数 > 512；listpack → hashtable 见配置 |
| List | quicklist（listpack + linkedlist） | - | 始终使用 quicklist |
| Zset | listpack | skiplist+hashtable | 元素数 > 128 或元素长度 > 64 字节 |

**为什么进行编码转换**：

核心思想是**用时间换空间**：
1. 数据量小时，使用紧凑编码（listpack/intset）节省内存。比如一个只有 3 个字段的 Hash，用 hashtable 需要分配哈希表结构 + 链表节点，开销远大于数据本身。
2. 数据量大时，紧凑编码的性能劣势（O(n) 查找、内存重分配）会放大，此时转换为标准数据结构保证操作效率。
3. 自动转换对用户透明，体现了 Redis "在存储效率和访问效率之间自动选择最优方案"的设计理念。

</details>

## 四、场景题（共 25 分）

### 题目 14（10 分）
**场景**：线上 Redis 中，一个 Hash 类型存储了用户的基本信息（name、age、city 等字段，约 10 个字段），值都是较短的字符串。运维同事建议将这个 Hash 拆分成多个 String 类型的 key 来存储（如 `user:1:name`、`user:1:age`、`user:1:city`），理由是"Hash 操作更灵活"。

请从**内存效率**和**编码方式**的角度分析：
1. 当前 Hash 的编码方式是什么？为什么？
2. 拆分成 String 后，内存占用是增加还是减少？请给出理由。

<details>
<summary>答案与解析</summary>

**1. 当前 Hash 的编码方式**：

编码为 **listpack**。因为：
- 字段数量约 10 个，远小于阈值 512
- 每个字段的键和值都是较短字符串，长度远小于 64 字节

在这个编码下，所有数据连续存储在一块内存中，几乎没有额外开销。

**2. 拆分成 String 后内存变化**：

内存占用会**显著增加**。原因：
- 每个 String 键都需要一个独立的 redisObject（16 字节）+ SDS 头部 + key 名称的 SDS 开销
- 10 个字段 = 10 个独立的 key，每个 key 有独立的 redisObject 和 dictEntry（哈希表桶节点，含 key/value/next 三个指针 = 24 字节）
- 而 listpack 编码的 Hash 只需要一个 key + 一个 redisObject + 连续的数据块

**结论**：对于小规模 Hash，保持 Hash 类型存储远比拆分为多个 String 节省内存。Redis 官方数据：小 Hash 比等量 String 节省约 10 倍内存。

</details>

### 题目 15（15 分）
**场景**：某电商系统使用 Redis Zset 存储商品排行榜。当前排行榜中有 1000 个商品，其中部分商品名称较长（超过 100 字节）。

请回答以下问题：

1. 该 Zset 的当前编码是什么？为什么？
2. 如果系统在秒杀活动中，排行榜的商品数量可能瞬间激增到 5000 个，编码是否会发生转换？转换过程是怎样的？
3. 从内存角度分析，如果排行榜只需要存储商品 ID 和分数，有什么优化建议？

<details>
<summary>答案与解析</summary>

**1. 当前编码**：

编码为 **skiplist + hashtable**（不是 listpack）。原因：
- 虽然元素数量 1000 > 128（阈值），已经触发转换
- 同时存在商品名称超过 64 字节，也不满足 listpack 的条件

Zset 使用双数据结构：skiplist 支持范围查询和排名操作，hashtable 提供 O(1) 的单元素查找。

**2. 数量激增时的编码变化**：

**不会发生转换**，因为当前已经是 skiplist + hashtable 编码，只会在大编码上继续增长。编码转换是单向的——从小编码转向大编码，不会反向转换。

如果当前是 listpack 编码（假设元素少于 128 且值都短于 64 字节），当元素超过 128 或值超过 64 字节时：
1. 创建 skiplist + hashtable 数据结构
2. 将 listpack 中所有元素逐一迁移
3. 释放旧的 listpack 内存
4. 编码标记改为 SKIPLIST

**3. 优化建议**：

- **使用数字 ID 代替长商品名称**：如果商品名称已经超过 64 字节，Zset 的 member 存储商品 ID（纯数字或短字符串），在需要展示名称时再从 Hash 或数据库中查询。这样既节省 Zset 的内存，也可能让小规模排行榜使用 listpack 编码。
- **考虑编码优化**：如果 member 全是整数且数量 <= 128、长度 <= 64 字节，Zset 会使用 listpack 编码，内存开销大幅减少。

</details>

## 附加题（共 20 分）

### 题目 16（10 分）
**Redis 的对象共享机制利用了 redisObject 的 `refcount` 字段。请说明：Redis 会共享哪些对象？为什么不大范围使用对象共享？**

<details>
<summary>答案与解析</summary>

**共享对象范围**：

Redis 只共享 **0 到 9999 的整数值字符串对象**。这些对象在 Redis 服务器启动时预先创建，存储在 `shared.integers` 数组中。当需要使用这些整数值时，直接引用共享对象的指针，`refcount` 设为 `OBJ_SHARED_REFCOUNT`（表示无限共享）。

**不大范围共享的原因**：

1. **CPU 代价过高**：共享对象要求判断两个对象是否"相等"。对于整数值，O(1) 可以判断；但对于字符串，需要 O(n) 逐字符比较；对于复杂对象（哈希、集合等），比较代价更高。
2. **内存收益不确定**：共享只节省了对象本身的内存，但需要维护引用计数。对于大对象，共享收益明显；但对于小对象，引用计数的开销可能抵消收益。
3. **LRU 淘汰策略冲突**：共享对象被多个 key 引用，无法确定"最后访问时间"。如果使用 LRU/LFU 淘汰策略，共享对象的 lru 字段无法准确反映每个 key 的访问情况。

**面试话术**：Redis 的对象共享是一种"用 CPU 换内存"的策略，但只对 0-9999 整数启用。因为整数的相等判断是 O(1)，而字符串需要 O(n) 比较，复杂度更高。这也解释了为什么 Redis 推荐用整数做 key 的前缀。

</details>

### 题目 17（10 分）
**请说明 ListPack 的 entry 结构设计，并解释为什么它能从根本上消除连锁更新问题。如果你面试时被问到"Redis 的压缩列表有什么问题"，你会怎么回答？**

<details>
<summary>答案与解析</summary>

**ListPack entry 结构**：
```
+----------+----------+----------+
| encoding |   data   | backlen  |
| 1-5 字节 |  变长    | 1-5 字节 |
+----------+----------+----------+
```

每个 entry 由三部分组成：
- **encoding**：记录数据类型和长度
- **data**：实际数据
- **backlen**：记录自身（encoding + data + backlen）的总长度，用于反向遍历

**消除连锁更新的原理**：

旧版 ZipList 的 entry 记录前一节点的长度（prevlen），当前一节点长度变化时，当前节点的 prevlen 也可能需要变化（1 字节 ↔ 5 字节），从而引发级联扩展。

ListPack 的 entry 记录的是自身长度（backlen），修改某个 entry 不会影响其他 entry 的存储格式，从根本上消除了连锁更新。

**面试回答模板**：

"ZipList 的连锁更新是因为 entry 记录前一节点长度，节点间形成耦合。Redis 7.0 用 ListPack 替代了 ZipList，entry 改为记录自身长度，实现了节点间解耦。在 Redis 7.0+ 中，Hash、ZSet、List 的底层紧凑编码已经全部使用 ListPack，连锁更新问题不再存在。"

</details>
