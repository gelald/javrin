## 高频面试题 Top 10

### 1. Redis 的持久化机制了解吗？RDB 和 AOF 有什么区别？（⭐ 最高频）

**标准回答（表格 + 结论）：**

Redis 提供两种持久化方式：

| 维度 | RDB | AOF |
|------|-----|-----|
| 原理 | 定时全量快照（二进制） | 追加每条写命令（文本） |
| 数据安全 | 可能丢最后一次快照后的数据 | 最多丢 1 秒（everysec） |
| 恢复速度 | 快（~10 倍） | 慢（逐条重放） |
| 文件大小 | 小（紧凑） | 大（需重写压缩） |
| 适用场景 | 冷备份、灾备 | 数据安全要求高 |

**结论**：生产环境建议**同时开启**，用 AOF 保数据安全，用 RDB 做冷备份和快速恢复。Redis 4.0+ 推荐使用混合持久化。

---

### 2. AOF 的三种 fsync 策略是什么？生产环境怎么选？（⭐ 高频）

**标准回答：**

| 策略 | fsync 频率 | 数据丢失风险 | 性能 |
|------|-----------|-------------|------|
| always | 每条命令 | 最多丢 1 条 | 最差 |
| **everysec** | **每秒** | **最多丢 1 秒** | **较好** |
| no | OS 决定 | 可能丢 30 秒 | 最好 |

**生产选择**：`everysec`，这是性能和数据安全的最佳平衡。always 性能太差（约为 everysec 的 1/10），no 丢数据太多。

---

### 3. AOF 重写是什么？重写期间有新写入怎么办？（⭐ 高频）

**标准回答（流程 + 关键细节）：**

AOF 重写是**直接遍历当前内存数据**，生成最精简的写命令，减小 AOF 文件体积。

**流程**：
1. fork 子进程，子进程基于内存快照生成新 AOF
2. 主进程继续处理请求，新写命令同时写入 **AOF 缓冲区** 和 **AOF 重写缓冲区**
3. 子进程完成后，主进程将重写缓冲区追加到新 AOF
4. 原子替换旧文件

**关键点**：重写缓冲区保证了 fork 后的增量数据不丢失。

---

### 4. 什么是混合持久化？（⭐ 中高频）

**标准回答：**

Redis 4.0 引入的方案。AOF 重写时先用 **RDB 格式**写入全量数据（体积小、加载快），再追加 **AOF 格式**的增量命令（保证完整性）。

恢复时先加载 RDB 部分（快），再重放 AOF 部分（完整），兼顾了两种方案的优点。配置 `aof-use-rdb-preamble yes` 开启。

---

### 5. fork 和 COW 是什么？为什么大内存实例会有问题？（⭐ 高频）

**标准回答（3 层递进）：**

**第一层 — fork**：bgsave 和 bgrewriteaof 都通过 fork() 创建子进程。fork 后父子进程共享物理内存（不是真的拷贝 10GB），只复制页表。

**第二层 — COW**：主进程修改数据时，操作系统才复制对应的内存页（写时复制），子进程继续用 fork 时的快照。

**第三层 — 大内存问题**：
- **fork 耗时与内存成正比**：30GB 实例 fork 可能要 1 秒，期间主线程完全阻塞
- **COW 内存膨胀**：fork 后写操作频繁，复制大量内存页，实际内存可能翻倍
- **建议**：单实例 < 10GB，大实例拆分

---

### 6. 生产环境 Redis 持久化方案怎么设计？（⭐ 场景题高频）

**标准回答（模板式）：**

```
# 开启 AOF + 混合持久化
appendonly yes
appendfsync everysec
aof-use-rdb-preamble yes

# RDB 低频快照做冷备
save 3600 1 300 100 60 10000
```

**补充说明**：
- 单实例内存 < 10GB，避免 fork 问题
- RDB 文件定期备份到远程存储
- 纯缓存场景可以只开 RDB 或关闭持久化
- 监控 fork 耗时和 fsync 延迟

---

### 7. AOF 为什么用"写后日志"而不是"写前日志（WAL）"？

**标准回答：**

三个原因：
1. **避免语法检查**：写前需要先验证命令语法；写后记录的是已成功执行的命令
2. **不阻塞当前命令**：先执行返回结果，再记录日志
3. **不记录错误命令**：执行失败的命令不会进入日志

**风险**：命令执行成功但还没来得及记录就宕机，会丢失这条命令。用 `everysec` 策略可将风险降到最低（最多丢 1 秒）。

---

### 8. RDB 和 AOF 同时开启时，Redis 重启用哪个恢复？

**标准回答：**

优先使用 **AOF 文件**恢复，因为 AOF 的数据完整性更高。RDB 文件作为兜底冷备份。这是 Redis 的默认行为。

---

### 9. 大 Key 对持久化有什么影响？怎么处理？

**标准回答：**

**影响**：
- 修改大 Key 触发大量内存页复制，加剧 COW 内存膨胀
- RDB/AOF 重写时序列化大 Key 耗时长

**处理**：
- 拆分大 Key（Hash 拆为多个小 Hash）
- 删除大 Key 用 `UNLINK`（异步，不阻塞主线程）
- `redis-cli --bigkeys` 定期扫描

---

### 10. 怎么监控 Redis 持久化的性能？

**标准回答：**

| 监控项 | 方法 | 关注点 |
|--------|------|--------|
| fork 耗时 | `latency latest` | > 100ms 需关注 |
| AOF fsync 延迟 | `latency latest` | > 10ms 需关注 |
| RDB 上次保存时间 | `lastsave` | 是否按时触发 |
| AOF 文件大小 | `info Persistence` | 是否需要重写 |
| 内存使用 | `info Memory` | 是否超过阈值 |

```bash
# 开启延迟监控
config set latency-monitor-threshold 100

# 查看最近的延迟事件
latency latest
```

## 易错点总结

### 1. AOF 重写不是"压缩旧 AOF 文件"

- 重写是**直接遍历内存**生成新文件
- 不是读取旧 AOF 再压缩
- 所以重写后的文件一定是最精简的

### 2. fork 不是"复制全部内存"

- fork 后父子进程共享物理内存
- 只复制页表（元数据）
- 实际内存复制发生在 COW（写时复制）

### 3. 混合持久化文件的结构顺序

- **前半部分是 RDB 格式**（全量快照）
- **后半部分是 AOF 格式**（增量命令）
- 不要搞反了

### 4. AOF everysec 不是"绝对不会丢数据"

- 最多丢 1 秒的数据
- 如果 fsync 线程正好在执行时宕机，可能丢 2 秒
- 但比 RDB 的"可能丢几分钟甚至几小时"好得多

### 5. bgsave 期间不是"完全不影响主线程"

- fork 瞬间主线程是阻塞的（复制页表）
- 只是 fork 之后主线程恢复正常
- 大内存实例的 fork 阻塞可能很明显

## 面试话术模板

### 对比句式

> RDB 和 AOF 的核心区别在于：RDB 是"拍照"——某一时刻的全量快照；AOF 是"录像"——记录每一条写命令。RDB 恢复快但可能丢数据，AOF 数据安全但恢复慢。生产环境一般同时开启。

### 原理句式

> COW 的全称是 Copy-On-Write，写时复制。fork 后父子进程共享物理内存，只有主进程修改某个内存页时，操作系统才复制这一页给主进程。所以 fork 本身很快（只复制页表），但 fork 之后的写操作会逐渐增加额外内存。

### 场景句式

> 我们生产环境用的是 RDB + AOF 混合持久化。AOF 用 everysec 策略保证数据安全，RDB 做冷备份。单实例控制在 10GB 以内避免 fork 问题，关闭了 Linux 透明大页优化 COW 表现。
