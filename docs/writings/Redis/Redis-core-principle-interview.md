## 高频面试题 Top 10

### 1. Redis 为什么用单线程？为什么还这么快？

**标准回答（3 层递进）：**

Redis 核心命令执行是单线程的，这是刻意的设计选择。原因有三层：

**第一层 — 瓶颈分析**：Redis 的数据全在内存中，内存操作是纳秒级的，CPU 从来不是瓶颈。真正的瓶颈在**网络 IO**。

**第二层 — 单线程优势**：
- **避免上下文切换**：线程切换需要保存/恢复寄存器、栈指针，开销不小
- **避免锁竞争**：单线程天然串行，不需要加锁，没有死锁、竞态条件
- **实现简单**：不需要考虑并发安全，代码逻辑清晰

**第三层 — 高性能组合拳**：
- **IO 多路复用**（epoll）：一个线程监听数万个连接
- **高效数据结构**：SDS、ZipList、SkipList 等
- **简单协议**：RESP 解析开销极低

**加分补充**：Redis 6.0 引入了 IO 多线程，但命令执行仍然是单线程的。说明 Redis 官方也认为瓶颈在网络 IO 而非 CPU。

---

### 2. 解释一下 IO 多路复用，select/poll/epoll 有什么区别？

**标准回答：**

IO 多路复用是指一个线程同时监听多个文件描述符（FD），哪个就绪就处理哪个。

三者核心区别：

| 维度 | select | poll | epoll |
|------|--------|------|-------|
| 连接数上限 | 1024 | 无限制 | 无限制 |
| 时间复杂度 | O(n) 遍历所有 FD | O(n) | O(1) 事件回调 |
| 数据拷贝 | 每次全量拷贝 | 每次全量拷贝 | 只拷贝就绪的 |
| 触发模式 | 水平触发 | 水平触发 | 支持 LT 和 ET |

**Redis 的选择**：Redis 封装了 `ae.c` 抽象层，Linux 用 epoll，macOS 用 kqueue，自动选择最优实现。

---

### 3. Redis 的事件驱动模型是怎么工作的？

**标准回答：**

Redis 基于 `ae.c` 实现了一个精简的事件驱动框架，处理两类事件：

- **文件事件**：客户端连接的读写事件（网络 IO）
- **时间事件**：定时任务，最核心的是 `serverCron`（每 100ms）

**事件循环流程**：
1. 查找最近的定时器，计算 epoll_wait 超时时间
2. 调用 epoll_wait 等待文件事件
3. 有文件事件就处理（读写客户端数据）
4. 检查时间事件是否到期，到期就执行

**关键源码**：`ae.c:aeProcessEvents()` 是事件处理的核心函数。

---

### 4. Redis 6.0 的多线程是怎么回事？能详细说说吗？

**标准回答（踩分点）：**

**核心结论**：Redis 6.0 的多线程**只用于网络 IO**，命令执行仍然是单线程。

**为什么引入**：当 QPS 超过 10 万时，网络 IO 读写成为瓶颈，单线程处理网络数据的耗时占比越来越大。

**工作方式**：
1. 多个 IO 线程**并行读取**客户端请求
2. 主线程**串行执行**所有命令（保证原子性）
3. 多个 IO 线程**并行写回**响应

**配置方式**：
```redis
io-threads 4              # IO 线程数
io-threads-do-reads yes   # 开启多线程读
```

**默认关闭**：小规模场景下单线程已足够，多线程有线程同步开销，需根据实际负载决定。

### 5. Redis 过期 Key 是怎么删除的？

**标准回答（双策略）：**

Redis 采用**惰性删除 + 定期删除**组合策略：

**惰性删除**（`expireIfNeeded`）：
- 每次访问 Key 时检查是否过期
- 过期就删除，未过期正常返回
- 优点：CPU 友好；缺点：不访问就不删除，可能内存泄漏

**定期删除**（`activeExpireCycle`）：
- 每 100ms 从过期字典中随机抽取 20 个 Key
- 过期比例超过 25% 就继续下一轮
- 有时间上限（默认 25ms）避免阻塞

**为什么用双策略**：单独用惰性会内存泄漏，单独用定期会遗漏。两者组合互补。

---

### 6. Redis 有哪些内存淘汰策略？怎么选择？

**标准回答：**

8 种策略，分为三大类：

| 类别 | 策略 | 说明 |
|------|------|------|
| 不淘汰 | `noeviction` | 默认，写入报错 |
| 全量淘汰 | `allkeys-lru` / `allkeys-lfu` / `allkeys-random` | 从所有 Key 中淘汰 |
| 过期淘汰 | `volatile-lru` / `volatile-lfu` / `volatile-random` / `volatile-ttl` | 只淘汰设了 TTL 的 Key |

**选择建议**：
- 大多数场景：**`allkeys-lru`**（最通用）
- 有明确冷热区分：**`allkeys-lfu`**
- Redis 同时存持久化数据：**`volatile-lru`**（保护不过期的数据）

---

### 7. Redis 的 LRU 是真正的 LRU 吗？说一下近似 LRU 算法

**标准回答：**

不是真正的 LRU，Redis 使用**近似 LRU**。

**为什么不用严格 LRU**：
- 严格 LRU 需要全局链表，每次访问都移动节点，CPU 开销大
- 需要额外内存维护链表节点

**近似 LRU 做法**：
1. 每个 Key 的 `redisObject.lru`（24 bit）记录最后访问时间
2. 淘汰时随机采样 N 个 Key（`maxmemory-samples`，默认 5）
3. 从采样中选最久未访问的淘汰
4. 重复采样直到释放足够内存

**效果**：官方测试表明，采样数为 5 时接近真实 LRU 90%+ 的效果，10 时几乎一致。

---

### 8. LRU 和 LFU 有什么区别？

**标准回答：**

| 维度 | LRU | LFU |
|------|-----|-----|
| 淘汰依据 | 最后访问时间 | 访问频率 |
| 适用场景 | 热点随时间变化 | 有稳定热点 |
| lru 字段 | 24 bit 存时间戳 | 高 16 位衰减时间 + 低 8 位计数器 |

**LFU 的两个关键机制**：
- **衰减**（`lfu-decay-time`，默认 1 分钟）：计数器随时间递减，避免历史热点永远不被淘汰
- **对数增长**（`lfu-log-factor`，默认 10）：计数器基于概率增长，防止溢出

---

### 9. 为什么 Redis 只共享整数对象，不共享字符串对象？

**标准回答：**

Redis 启动时预创建 0 ~ 9999 的整数对象共享池。

**只共享整数的原因**：
- 整数比较是 O(1)，可以直接判断两个值是否相等
- 字符串比较是 O(n)，需要逐字符比对，查找共享池的成本太高

**不共享字符串的原因**：
- 字符串可能很长，逐个比较性能太差
- 字符串值无穷多，共享池大小无法控制

### 10. 什么是 Reactor 模式？Redis 是怎么用的？

**标准回答：**

Reactor 模式是一种事件驱动的并发处理模式，核心思想是：**由一个事件循环线程监听所有 IO 事件，事件就绪时分发给对应的处理器**。

Redis 使用的是**单 Reactor 单线程**模型：

```
epoll_wait 监听所有连接
    ↓ 有事件就绪
分发器根据事件类型分发
    ↓ 读事件                    ↓ 写事件
readQueryFromClient()      sendReplyToClient()
    ↓
processCommand() 执行命令
```

优点：简单、无锁、高效。缺点：无法利用多核 CPU（Redis 6.0 通过 IO 多线程部分解决）。

## 易错点总结

### 1. "Redis 是单线程的" 不完全准确

- 核心命令执行是单线程
- 但 Redis 还有**后台线程**处理：BGSAVE（fork 子进程）、AOF 重写、异步删除（UNLINK）等
- Redis 6.0 还有 IO 多线程

### 2. 定期删除不是"每隔固定时间扫描全部 Key"

- 是**随机抽样**，每轮只抽 20 个
- 不是全量扫描，不可能覆盖所有 Key
- 所以才会有惰性删除来兜底

### 3. Redis 6.0 多线程 ≠ 命令并行执行

- 多线程只负责网络 IO（读请求、写响应）
- 命令执行仍在主线程串行进行
- 所以 MULTI/EXEC、Lua 脚本的原子性不受影响

### 4. noeviction 不是"内存满了就宕机"

- 是**拒绝所有写入操作**（返回错误）
- 读取操作正常
- 不会丢失已有数据

### 5. LFU 的计数器不是简单的 +1

- 使用基于对数的概率增长算法
- 避免热点 Key 计数器溢出
- 配合衰减机制，保证历史数据不会永远占位

## 面试话术模板

### 开场句式

> Redis 的核心设计理念是：**用简单的方式解决复杂的问题**。单线程模型 + IO 多路复用就是最好的体现。

### 对比句式

> 对比 select 和 epoll：select 每次调用要遍历所有 FD，O(n) 复杂度；epoll 是事件驱动，O(1) 复杂度。在万级并发下差异巨大。

### 总结句式

> 简单来说，Redis 快不是因为单线程本身快，而是因为**瓶颈不在 CPU，而在网络 IO**。单线程避免了锁和上下文切换的开销，配合 IO 多路复用，用最简单的方式达到了最高的性能。
