## 高频面试题 Top 10

### 1. 缓存穿透、缓存击穿、缓存雪崩分别是什么？怎么解决？（⭐ 最高频）

**标准回答（一句话区分 + 分别展开）**：

**一句话区分**：
- **穿透** — 数据不存在（缓存和 DB 都查不到）
- **击穿** — 热点过期（大量请求打到 DB）
- **雪崩** — 大面积过期或 Redis 宕机

**分别展开**：

| 问题 | 解决方案 | 生产推荐 |
|------|---------|---------|
| 穿透 | 布隆过滤器 + 缓存空对象 | 布隆过滤器前置 + 参数校验 |
| 击穿 | 互斥锁 / 逻辑过期 | 读多场景用逻辑过期，强一致场景用互斥锁 |
| 雪崩 | TTL 随机化 + 多级缓存 + 高可用 | TTL 加随机值 + 本地缓存兜底 + Redis 哨兵 |

**加分回答**：实际项目中通常是多种方案组合使用。比如商品详情页：布隆过滤器防穿透 + TTL 随机化防雪崩 + 互斥锁防击穿 + Caffeine 本地缓存做二级兜底。

### 2. 如何保证缓存与数据库的双写一致性？（⭐ 最高频）

**标准回答（方案递进）**：

**基础方案 — Cache Aside + 先更新 DB 后删缓存**：

这是业界标准做法。写操作先更新数据库再删除缓存。为什么"删除"而不是"更新"？因为懒加载减少无效写。

**可能不一致的场景**：

极端并发下（读 DB 耗时 > 写 DB + 删缓存），可能出现缓存中是旧值。但这个概率极低。

**进阶方案 — 延迟双删**：

```
1. 删除缓存
2. 更新数据库
3. 延迟 200-500ms 后再次删除缓存
```

延迟时间 > 一次读操作的总耗时，确保可能写入旧值的线程已经完成。

**最佳方案 — Canal 监听 Binlog**：

```
DB 写入 → Binlog → Canal 解析 → 异步删除缓存
```

对业务代码零侵入，可靠性高，是大型系统的首选方案。

### 3. 布隆过滤器了解吗？原理是什么？有什么缺点？（⭐ 高频）

**标准回答**：

布隆过滤器是一种空间高效的概率型数据结构，用于判断一个元素**是否可能存在**。

**原理**：
1. 使用一个位数组（bit array）和多个哈希函数
2. 插入元素时，用 K 个哈希函数计算出 K 个位置，将对应位设为 1
3. 查询时，检查 K 个位置是否都为 1：如果有任何一个为 0，则元素一定不存在；如果都为 1，则元素**可能存在**（有误判率）

**缺点**：
- **误判率**：可能误报存在（false positive），但不会漏报（没有 false negative）
- **不支持删除**：删除一个元素可能影响其他元素。可用 Counting Bloom Filter 解决
- **容量固定**：创建后无法扩容

**适用场景**：已知数据集合、读多写少、能容忍少量误判。

### 4. Redis 分布式锁怎么实现？有什么问题？（⭐ 最高频）

**标准回答（三段式）**：

**第一段 — 基本实现**：

```
加锁: SET lock_key unique_value NX EX 10
解锁: Lua 脚本（先 GET 判断是否自己的锁，再 DEL）
```

NX 保证互斥，EX 防止死锁，unique_value + Lua 防止误删。

**第二段 — 存在的问题**：

| 问题 | 说明 |
|------|------|
| 锁过期释放 | 业务没执行完锁就过期了 |
| 不可重入 | 同一线程不能多次获取同一把锁 |
| 集群主从切换 | 主节点加锁后宕机，从节点升主后锁丢失 |

**第三段 — 引出 Redisson**：

生产环境推荐使用 Redisson，它解决了上述所有问题：
- **看门狗**：自动续期，防止锁过期
- **Hash 结构**：支持可重入
- **Pub/Sub**：锁释放通知，避免自旋空转
- **Lua 脚本**：所有操作原子化

### 5. Redisson 的看门狗机制是怎么工作的？（⭐ 高频）

**标准回答**：

看门狗（Watchdog）解决的问题是：业务执行时间超过锁的过期时间，锁自动释放导致并发问题。

**工作机制**：
1. 调用 `lock()` 不指定 leaseTime 时自动启动后台定时线程
2. 每隔 `lockWatchdogTimeout/3`（默认 10 秒）执行一次
3. 检查锁是否仍被当前线程持有，如果是则重置过期时间为 `lockWatchdogTimeout`（默认 30 秒）
4. 客户端宕机 → 看门狗线程终止 → 锁自动过期释放

**关键点**：指定了 leaseTime（如 `tryLock(5, 30, TimeUnit.SECONDS)`）则**不会启动看门狗**，锁在到期后一定会释放。

### 6. 项目中分布式锁怎么用的？遇到过什么问题？（⭐ 场景题高频）

**标准回答模板**：

> 在我们项目中，分布式锁主要用于以下场景：
>
> 1. **库存扣减**：防止超卖。多个服务实例同时扣减库存时，用分布式锁保证原子性
> 2. **定时任务**：防止多实例重复执行。分布式部署的定时任务加锁，只有一个实例执行
> 3. **订单状态变更**：防止并发修改
>
> 技术选型上使用 **Redisson**，主要原因：
> - 可重入锁支持嵌套调用
> - 看门狗自动续期，不用担心业务执行时间超过锁过期时间
> - Lua 脚本保证原子性
>
> 遇到的问题：
> - 早期用手写 SET NX + EX，出现过锁过期后业务没执行完的问题，后来切换到 Redisson
> - 锁粒度太大导致并发度低，后来改为细粒度锁（如锁到订单 ID 级别）

### 7. 大 Key 问题了解吗？怎么发现和处理？（⭐ 高频）

**标准回答**：

**什么是大 Key**：Value 过大或元素过多的 Key。一般 String > 10KB，集合类型元素 > 5000 就算大 Key。

**危害**：
- 集群内存倾斜（数据分布不均）
- 操作大 Key 阻塞主线程（DEL、HGETALL 等耗时过长）
- 持久化时 fork 耗时、COW 内存膨胀

**发现方法**：
```bash
redis-cli --bigkeys          # 扫描大 Key
redis-cli memory usage key   # 查看单个 Key 内存占用
```

**处理方案**：
| 方案 | 说明 |
|------|------|
| 拆分 | 大 Hash 拆为多个小 Hash |
| 压缩 | 大 String 用 gzip 压缩 |
| 异步删除 | `UNLINK` 替代 `DEL` |
| 本地缓存 | 大 Key 放本地，只缓存摘要到 Redis |

### 8. Redis 做限流有哪几种方案？（⭐ 高频）

**标准回答（对比表格）**：

| 算法 | 原理 | Redis 实现 | 特点 |
|------|------|-----------|------|
| 固定窗口 | 统计固定时间段内请求数 | INCR + EXPIRE | 简单但有临界突发问题 |
| 滑动窗口 | 细化时间窗口，滑动统计 | ZSET（score=时间戳） | 更平滑 |
| 漏桶 | 恒定速率流出 | List + 定时消费 | 平滑但无法处理突发 |
| **令牌桶** | 匀速生成令牌，允许突发消耗 | **Lua 脚本（推荐）** | **允许突发+限流** |

**生产推荐**：令牌桶算法（Lua 脚本实现），兼顾限流和突发处理能力。

### 9. 为什么缓存写操作推荐"删除"而不是"更新"？（⭐ 中高频）

**标准回答**：

三个原因：
1. **懒加载效率**：如果频繁更新的 Key 很少被读取，"更新缓存"浪费性能。"删除"只在下次读取时才重建，按需加载
2. **并发安全**：并发更新同一个 Key 时，"更新缓存"可能导致缓存和 DB 的版本不一致（两个写操作都成功更新了缓存，但顺序可能与 DB 不一致）
3. **复杂度低**："删除"是一次操作，"更新"需要计算差值或全量覆盖，逻辑更复杂

**例外场景**：如果某个 Key 读取频率极高且更新频率也高，可以考虑"更新缓存"以避免重建期间的短暂 miss。

## 易错点总结

### 1. 混淆穿透、击穿、雪崩

这是面试最高频的区分题。记住：
- **穿透** = 数据不存在（布隆过滤器解决）
- **击穿** = 热点过期（互斥锁/逻辑过期解决）
- **雪崩** = 大面积过期（TTL 随机化解决）

### 2. "先删缓存后更新 DB" 的问题

很多人以为先删缓存后更新 DB 没问题。实际上这种顺序更容易导致不一致：线程 A 删缓存后、更新 DB 前，线程 B 读缓存 miss 并将旧值写入缓存。

### 3. 看门狗和 leaseTime 的关系

指定了 leaseTime 就不会启动看门狗。面试常考"什么情况下启动看门狗"，答 `lock()` 无参时启动，`tryLock(wait, lease, unit)` 不启动。

### 4. Redisson 的锁不是强一致的

Redis 主从架构下，主节点加锁后宕机且未同步到从节点，从节点升主后锁丢失。Redisson 单节点锁不保证强一致性，只是"足够可靠"。如果需要强一致性，应该用 ZooKeeper/etcd。

### 5. 大 Key 删除用 UNLINK 不是 DEL

`DEL` 是同步删除，大 Key 会阻塞主线程。`UNLINK` 是 Redis 4.0+ 提供的异步删除命令，后台线程执行，不阻塞主线程。

### 6. Pipeline 不保证原子性

Pipeline 只是减少 RTT 的批量发送，不保证命令之间的原子性。如果需要原子性，使用事务（MULTI/EXEC）或 Lua 脚本。

### 7. 缓存空对象的 TTL 不能太长

缓存空对象时 TTL 不能设置太长（建议 60 秒以内），否则恶意攻击者可以用大量不同的不存在的 ID 占满缓存空间，导致正常数据被淘汰。

## 经典面试话术模板

### 模板一："缓存一致性怎么做？"（完整回答）

> 我们项目用 **Cache Aside + 延迟双删** 方案：
>
> 1. 写操作：先更新数据库，再删除缓存，延迟 500ms 后再删一次
> 2. 读操作：先查缓存，miss 时查数据库并写入缓存，TTL 加随机值防雪崩
>
> 对于一致性要求更高的场景，我们通过 **Canal 监听 MySQL Binlog**，数据变更后异步删除缓存。这个方案对业务代码零侵入。
>
> 需要注意的是，Cache Aside 是最终一致方案，不保证强一致。如果业务要求强一致，需要用分布式读写锁或者直接查数据库。

### 模板二："分布式锁怎么实现？"（30 秒简版）

> 基本实现是 `SET lock_key unique_value NX EX` 加锁，Lua 脚本判断 + 删除解锁。生产环境用 **Redisson**，解决了三个问题：看门狗自动续期防止锁过期、Hash 结构支持可重入、Pub/Sub 通知避免自旋空转。要注意指定 leaseTime 时不会启动看门狗。

### 模板三："缓存三大问题怎么防？"（场景式回答）

> 我们项目做了多层防护：
>
> 1. **穿透**：入口参数校验 + 布隆过滤器，拦截明显非法的请求
> 2. **击穿**：互斥锁 + 短期空值缓存，保证只有一个线程查库重建
> 3. **雪崩**：TTL 加随机值 + Caffeine 本地缓存做二级兜底 + Redis 哨兵保证高可用
>
> 实际上线后没有出现过缓存相关的问题。
