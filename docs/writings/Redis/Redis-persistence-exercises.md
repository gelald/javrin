## 一、选择题（每题 3 分，共 30 分）

### 1. 关于 Redis RDB 持久化，以下说法错误的是？

A. bgsave 通过 fork 子进程来完成快照，不阻塞主线程
B. RDB 文件是二进制格式，恢复速度比 AOF 快
C. RDB 可以做到秒级的数据安全保证
D. RDB 文件体积比 AOF 文件小

<details>
<summary>答案</summary>

**C**。RDB 是定时快照，无法做到秒级数据安全。两次快照之间的数据可能全部丢失。AOF 的 everysec 策略才能做到最多丢 1 秒数据。
</details>

---

### 2. AOF 持久化的三种 fsync 策略中，生产环境推荐使用哪种？

A. always
B. everysec
C. no
D. 由操作系统决定

<details>
<summary>答案</summary>

**B**。`everysec` 每秒 fsync 一次，是性能和数据安全的最佳平衡点。`always` 性能太差，`no` 可能丢太多数据。
</details>

---

### 3. AOF 重写时，fork 之后主进程收到的新写命令会怎么处理？

A. 直接丢弃，等重写完成后再重新执行
B. 只写入旧 AOF 文件
C. 同时写入 AOF 缓冲区和重写缓冲区
D. 阻塞直到重写完成

<details>
<summary>答案</summary>

**C**。fork 之后的新写命令会同时追加到 AOF 缓冲区（保证正常 AOF 写入）和 AOF 重写缓冲区（保证重写完成后合并增量数据）。
</details>

---

### 4. 关于 fork 和 COW，以下描述正确的是？

A. fork 后子进程会立即复制父进程的全部物理内存
B. COW 是"写前复制"，修改前先复制整个内存页
C. fork 的耗时与 Redis 实例的内存大小成正比
D. COW 机制不会增加额外内存开销

<details>
<summary>答案</summary>

**C**。fork 需要复制页表，页表大小与内存成正比。A 错：fork 后共享物理内存，不立即复制。B 错：COW 是"写时"复制，不是"写前"。D 错：COW 会在主进程修改时复制内存页，增加内存开销。
</details>

---

### 5. Redis 4.0 的混合持久化，AOF 重写后的文件结构是？

A. 全部是 AOF 格式的文本命令
B. 全部是 RDB 格式的二进制数据
C. 前半部分 RDB 格式 + 后半部分 AOF 格式
D. 前半部分 AOF 格式 + 后半部分 RDB 格式

<details>
<summary>答案</summary>

**C**。混合持久化的 AOF 文件先写入 RDB 格式的全量快照（体积小、加载快），再追加 AOF 格式的增量命令（保证数据完整性）。
</details>

---

### 6. 以下哪种情况会导致 AOF 文件最大？

A. 只做了少量写操作
B. 对同一个 Key 反复执行了上万次 SET 操作且从未重写
C. 使用了 everysec 的 fsync 策略
D. 开启了混合持久化

<details>
<summary>答案</summary>

**B**。AOF 记录每条写命令，对同一个 Key 反复执行上万次 SET 会记录上万条命令，如果不重写，文件会非常大。
</details>

---

### 7. 关于 `save` 命令，以下描述正确的是？

A. save 是推荐的 RDB 触发方式
B. save 在后台异步执行，不影响主线程
C. save 在主线程中同步执行，会阻塞所有客户端请求
D. save 和 bgsave 的执行机制完全相同

<details>
<summary>答案</summary>

**C**。`save` 在主线程中同步执行 RDB 快照，期间所有客户端请求被阻塞，生产环境禁用。推荐使用 `bgsave`。
</details>

---

### 8. 一个 20GB 内存 的 Redis 实例，fork 大约需要多长时间（Linux 物理机参考值）？

A. < 10ms
B. 10-100ms
C. 100ms-1s
D. > 5s

<details>
<summary>答案</summary>

**C**。10-30GB 实例 fork 耗时约 100ms-1s，期间主线程被阻塞。这就是为什么建议单实例内存控制在 10GB 以内。
</details>

---

### 9. 当 Redis 同时开启了 RDB 和 AOF，重启时使用哪个文件恢复数据？

A. 使用 RDB 文件，因为恢复更快
B. 使用 AOF 文件，因为数据更完整
C. 随机选择一个
D. 两个都加载，取最新数据

<details>
<summary>答案</summary>

**B**。Redis 官方文档明确：当 AOF 和 RDB 都开启时，重启优先使用 AOF 文件恢复数据，因为 AOF 的数据完整性更高。
</details>

---

### 10. 以下哪种方式可以减少 fork 对 Redis 主线程的影响？

A. 增大 Redis 实例内存
B. 使用虚拟机部署 Redis
C. 控制单实例内存 < 10GB，关闭 Linux 大页（THP）
D. 使用 save 命令替代 bgsave

<details>
<summary>答案</summary>

**C**。控制内存大小减少 fork 耗时，关闭透明大页（THP）避免加剧 COW 问题。A 反而会增大 fork 耗时。B 虚拟机的 fork 更慢。D save 更差，完全阻塞主线程。
</details>

## 三、简答题（每题 10 分，共 30 分）

### 1. 请对比 RDB 和 AOF 的优缺点，至少各说 3 点。

<details>
<summary>答案</summary>

**RDB 优点**：
1. 恢复速度快（直接加载二进制文件，约为 AOF 的 10 倍）
2. 文件体积小（二进制压缩格式）
3. 对性能影响小（fork 子进程干活，主进程基本不受影响）
4. 适合冷备份和灾备

**RDB 缺点**：
1. 无法实时持久化，可能丢失最后一次快照后的所有数据
2. 大内存实例 fork 耗时较长

**AOF 优点**：
1. 数据安全性高（everysec 最多丢 1 秒）
2. 文件可读性好（文本格式，可人工修复）
3. 支持重写压缩

**AOF 缺点**：
1. 恢复速度慢（逐条重放命令）
2. 文件体积大（需要定期重写）
3. 持续的 IO 写入对性能有一定影响
</details>

---

### 2. 请描述 AOF 重写的完整流程，重点说明重写期间有新写入时如何保证数据不丢失。

<details>
<summary>答案</summary>

1. **触发重写**：手动 `BGREWRITEAOF` 或自动触发（文件增长超过阈值）
2. **fork 子进程**：主进程调用 fork()，子进程获得内存快照
3. **子进程生成新 AOF**：遍历当前内存数据，生成最精简的写命令，写入临时文件
4. **主进程继续工作**：
   - 正常的写命令照常追加到 **AOF 缓冲区**（写入当前 AOF 文件）
   - **同时**追加到 **AOF 重写缓冲区**（保证不丢增量数据）
5. **子进程完成**：通知主进程重写完成
6. **合并增量**：主进程将重写缓冲区中的增量命令追加到新 AOF 文件末尾
7. **原子替换**：用 `rename()` 系统调用原子替换旧 AOF 文件

**关键保障**：重写缓冲区确保 fork 之后的所有新写命令都被保留，不会因为重写而丢失。
</details>

---

### 3. 什么是 COW（写时复制）？为什么大内存 Redis 实例的 fork 会影响性能？

<details>
<summary>答案</summary>

**COW 原理**：
- fork() 后父子进程共享同一块物理内存（不是真的拷贝）
- 只有当主进程需要**修改**某个内存页时，操作系统才复制该页给主进程
- 子进程始终使用 fork 时的内存快照，不受影响

**大内存实例的 fork 影响性能的原因**：
1. **fork 耗时与内存成正比**：fork 需要复制父进程的**页表**（Page Table），10GB 实例的页表复制可能需要数百毫秒
2. **fork 期间主线程阻塞**：fork 是同步操作，主线程在 fork 完成前无法处理任何请求
3. **COW 内存膨胀**：fork 后主进程的写操作越多，复制的内存页越多，额外内存开销越大

**建议**：生产环境单实例内存控制在 10GB 以内，大实例拆分为多个小实例。
</details>

## 四、场景题（每题 10 分，共 20 分）

### 1. 你们公司有一个 30GB 内存的 Redis 实例，最近频繁出现以下问题：

- 每次自动 bgsave 时，Redis 出现数百毫秒的延迟抖动
- 运维发现 Redis 进程的内存使用有时会飙到 50GB 以上

请分析原因并给出解决方案。

<details>
<summary>答案</summary>

**原因分析**：

1. **延迟抖动**：30GB 实例 fork 复制页表耗时较长（可能 1 秒+），期间主线程被阻塞，客户端请求超时

2. **内存飙到 50GB**：fork 后 COW 机制生效。bgsave 期间如果业务写操作频繁，大量内存页被复制，导致实际物理内存远超 30GB

**解决方案**：

1. **拆分实例**：将 30GB 拆分为 3-4 个 8GB 左右的小实例（最根本的解决方案）
2. **降低 fork 频率**：适当调大 `save` 配置的时间间隔，减少 bgsave 触发频率
3. **关闭大页**：`echo never > /sys/kernel/mm/transparent_hugepage/enabled`
4. **监控 fork 耗时**：开启延迟监控 `config set latency-monitor-threshold 100`
5. **使用物理机**：避免在虚拟机上运行大内存 Redis 实例
</details>

---

### 2. 你负责设计一个电商系统的 Redis 持久化方案，需求如下：

- 购物车数据不能丢失（用户加购后宕机需恢复）
- 商品缓存可以丢失（从数据库重新加载即可）
- Redis 版本为 7.0

请给出你的持久化配置方案，并说明理由。

<details>
<summary>答案</summary>

**推荐方案：混合持久化（RDB + AOF）**

```
# 开启 AOF
appendonly yes

# fsync 策略选择 everysec（性能与安全平衡）
appendfsync everysec

# 开启混合持久化
aof-use-rdb-preamble yes

# RDB 配置（作为冷备份，降低频率）
save 3600 1 300 100 60 10000

# 自动 AOF 重写配置
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
```

**理由**：
1. **AOF everysec**：保证购物车数据最多丢 1 秒，满足"不能丢"的需求
2. **混合持久化**：AOF 重写时先写 RDB 全量数据再追加增量，恢复时先加载 RDB（快），再重放少量 AOF（完整），兼顾恢复速度和数据完整性
3. **RDB 作为冷备**：即使 AOF 文件损坏，还有 RDB 可以恢复大部分数据
4. **商品缓存不需要特别处理**：丢失后从数据库重新加载即可，持久化方案自动覆盖
</details>

## 五、附加题（每题 10 分，共 20 分）

### 1. Redis AOF 为什么采用"写后日志"而不是"写前日志（WAL）"？写后日志有什么风险？

<details>
<summary>答案</summary>

**为什么用写后日志**：
1. **避免语法检查开销**：写前日志需要先验证命令语法是否正确再记录；写后日志记录的是已成功执行的命令，天然正确
2. **不阻塞当前命令**：先执行命令返回结果，再异步记录日志，不影响命令执行速度
3. **不会记录错误命令**：命令执行失败则不记录，日志中只有成功执行的命令

**写后日志的风险**：
1. **数据丢失**：命令执行成功但还没来得及写入 AOF 就宕机，这条命令会丢失
2. **主线程阻塞风险**：AOF 写入也是主线程执行的（写入 aof_buf），如果上一次 fsync 还没完成且策略是 always，可能阻塞主线程

**缓解措施**：使用 `everysec` 策略 + AOF 重写缓冲区，将风险降到最低。
</details>

### 2. 如果面试官问："你们生产环境的 Redis 是怎么做持久化的？如果让你来设计，你会怎么做？"

请给出你的回答（模拟面试场景）。

<details>
<summary>答案</summary>

**参考回答**：

我们生产环境使用的是 **RDB + AOF 混合持久化方案**。

**具体配置**：
- 开启 AOF，fsync 策略用 everysec，最多丢 1 秒数据
- 开启混合持久化（`aof-use-rdb-preamble yes`），AOF 重写时先写 RDB 再追加增量，恢复快
- RDB 配置为低频快照（如 `save 3600 1`），作为冷备份兜底

**实例规划**：
- 单实例内存控制在 8-10GB 以内，避免 fork 耗时过长
- 大实例拆分为多个小实例
- 关闭 Linux 透明大页（THP），减少 COW 影响

**监控与运维**：
- 开启延迟监控，关注 fork 耗时和 AOF fsync 延迟
- 定期检查 AOF 重写频率和文件大小
- RDB 文件定期备份到远程存储（如 S3），做灾备

**加分点**：如果是纯缓存场景（数据都在数据库），我会选择只开 RDB 甚至关闭持久化，减少性能损耗。
</details>
